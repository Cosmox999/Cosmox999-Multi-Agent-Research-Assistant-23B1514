## 📋 Overview

A production-quality **multi-agent AI research pipeline** built with LangGraph and Groq that autonomously researches a topic, analyses findings, critiques quality, and generates a structured final report — all in under 25 seconds.

**Demo Query:**
> *"Identify the top 5 emerging GenAI use cases in FMCG supply chains. For each use case, describe implementation approach, expected impact, risks, and maturity level."*

**Pipeline Results:**
- 🌐 **19 web sources** retrieved and deduplicated
- 🔁 **2 revision cycles** triggered by the critic agent
- ⭐ **Critic score: 8/10** on final output
- ⚡ **22.5 seconds** end-to-end runtime

## 🏗️ Architecture

```
User Query
    │
    ▼
┌─────────────┐     generates     ┌──────────────────────┐
│   PLANNER   │ ──────────────►  │  5 Search Queries    │
└─────────────┘                   └──────────────────────┘
                                            │
                                            ▼
┌─────────────┐   Tavily API    ┌──────────────────────┐
│ RESEARCHER  │ ◄────────────►  │  19 Web Sources      │
└─────────────┘                 └──────────────────────┘
       │
       ▼
┌─────────────┐
│   ANALYST   │ ◄─────────────────────────────┐
└─────────────┘                               │  revision loop
       │                                      │  (if score < 8)
       ▼                                      │
┌─────────────┐   score < 8   ┌──────────────┴─────┐
│    CRITIC   │ ─────────────► │  Feedback + Score  │
└─────────────┘                └────────────────────┘
       │ score ≥ 8
       ▼
┌─────────────┐
│  REPORTER   │ ──► final_report.md
└─────────────┘
```

All inter-agent communication flows through a single shared **TypedDict state object** managed by LangGraph's `StateGraph`.

## 🚀 Quick Start

### 1. Clone the repository
```bash
git clone https://github.com/Cosmox999/Multi-Agent-Research-Assistant.git
cd Multi-Agent-Research-Assistant
```

### 2. Install dependencies
```bash
pip install -r requirements.txt
```

### 3. Set up environment variables
Create a `.env` file in the root directory:
```env
TAVILY_API_KEY=your_tavily_api_key_here
GROQ_API_KEY=your_groq_api_key_here

# Optional
LOG_LEVEL=INFO   # Set to DEBUG for verbose output
```

> Get your free API keys: [Tavily](https://app.tavily.com) · [Groq](https://console.groq.com)

### 4. Run the pipeline
```bash
python main.py
```

### 5. Test error handling
```bash
python demo_edge_case.py
```

## 🧠 Agent Roles

| Agent | Responsibility | Output |
|-------|---------------|--------|
| **Planner** | Decomposes the research question into 5 targeted search queries | `search_queries: list` |
| **Researcher** | Runs parallel Tavily searches, deduplicates results, summarises with LLM | `sources: list`, `research: str` |
| **Analyst** | Produces structured use-case analysis; revises based on critic feedback | `analysis: str` |
| **Critic** | Scores output quality (0–10), flags specific issues, routes for revision | `critique_score: int`, `critique_approved: bool` |
| **Reporter** | Formats and writes the final deliverable report | `final_report: str` |

---

## 🔄 Critique-Revision Loop

The pipeline includes an **automatic quality-control loop**:

1. After the Analyst produces output, the **Critic** evaluates it and assigns a score
2. If `score < threshold (8)` and `revision_count < MAX_REVISIONS`, the graph routes **back to the Analyst** with specific feedback
3. The Analyst revises its output incorporating the critic's issues
4. Once approved (`score ≥ 8`) or revision cap is hit, the **Reporter** generates the final report

This loop ran **2 revision cycles** in the demo run before approval.

---

## 🛡️ Error Handling & Edge Cases

The pipeline is designed for graceful degradation:

- **Search failures** — If a Tavily query times out or errors, a warning is logged and the pipeline continues with remaining sources
- **JSON parse failures** — If the Analyst returns malformed JSON, raw text is stored as a fallback (observed and handled in revision 1)
- **Rate limiting** — A 3-second pause is inserted after the search phase to prevent downstream Groq API rate-limit errors
- **Infinite loop prevention** — `revision_count` is tracked and capped at `MAX_REVISIONS`

To see graceful degradation in action:
```bash
python demo_edge_case.py
# Simulates a ConnectionError on the 3rd search query
# Pipeline continues with ~16 sources instead of 20
```

## ⚠️ Known Limitations

- **Score ceiling at 8/10** — The revision prompt doesn't structurally inject specific critic issues as constraints; improvement possible with a JSON-schema-driven revision format
- **Context window limits** — Llama-3.1-8B has an 8K token window; very large source sets may be truncated (mitigated by per-query `max_results=4`)
- **No persistent memory** — Each run starts fresh; a vector store (e.g. ChromaDB) could enable incremental research across sessions
- **Hallucination risk** — Statistical claims generated without explicit citation grounding; future fix: enforce citation-required output schema
